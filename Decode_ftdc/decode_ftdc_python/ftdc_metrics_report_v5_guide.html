MongoDB FTDC / Python CLI

One script that decodes raw FTDC and turns it into a human-readable report.

`ftdc_metrics_report.py` replaces the awkward `bsondump`-plus-postprocessing flow with a single standard-library Python CLI. It reads raw `metrics.*` archives, `diagnostic.data` directories, or pre-decoded JSON snapshots, then emits a concise Markdown report focused on ops, latency, memory, CPU, and WiredTiger cache behavior.

Signature idea
3 inputs → 1 report

The script unifies decode, normalization, summarization, and presentation into one path, while preserving the enhanced metrics that were added on top of the original `report.py` / `report_1.py` lineage.

Execution paths

Two decode routes, one summary model.

Both paths converge into the same row-based summary engine. Use the buttons to highlight the path you care about.

Raw input

`metrics.*` or `diagnostic.data`

  • sorts `metrics.*` files
  • skips `metrics.interim`
  • treats the directory as one archive stream
Decode

Minimal BSON + zlib

  • `iter_bson_documents_from_file` reads outer BSON docs
  • type 1 chunks are inflated from `data[4:]`
  • varint delta stream is reconstructed into sample rows
Normalize

Target-path extraction

  • keeps `serverStatus` counters
  • preserves `systemMetrics.cpu.*` and `memory.*` aliases
  • flattens only the fields the report needs
Summarize

Shared report engine

  • builds interval statistics
  • computes deltas, rates, and normalized CPU
  • prints a Markdown report via `print_report`
Decoded input

Concatenated JSON snapshots

  • accepts `.json` directly
  • or `--decoded-json` to force this mode
  • useful for already-expanded FTDC dumps
Parse

Robust JSON stream reader

  • `parse_decoded_json_snapshots` consumes concatenated objects/arrays
  • stops cleanly on truncated tails
  • normalizes EJSON numeric wrappers
Project

Same row schema as raw path

  • `dig`, `dig_dotted`, and `num_ejson` map nested snapshot fields
  • decoded path matches the raw summarizer’s contract
Output

Support-ready Markdown

  • easy to paste into tickets, notes, or reviews
  • preserves enhanced latency and hardware sections
Function map

The script is organized into five functional layers.

These are the real components surfaced from the current `ftdc_metrics_report.py`, grouped by the job they do.

1 · formatting & helpers

Turns raw counters into readable values.

ms_to_iso fmt_duration fmt_bytes series_stats delta_from_series _latency_ms

These helper functions standardize timestamps, units, averages, deltas, and display formatting before the report is rendered.

2 · lookup & alias resolution

Matches FTDC field variants safely.

_normalize_lookup_key _normalized_mapping_lookup first_present first_present_scaled pct_series_from_counter normalize_pct_series

This is what lets CPU and memory metrics survive field-name differences across environments, especially under `systemMetrics.cpu.*` and `systemMetrics.memory.*`.

3 · raw FTDC decoding

Reconstructs compressed type 1 chunks without `bsondump`.

_read_cstring _decode_bson_document _decode_bson_value iter_bson_documents_from_file _signed_from_varint _read_varint _flatten_metric_schema iter_chunk_target_rows

This layer is the core differentiator: it handles outer BSON parsing, zlib decompression, schema flattening, and delta reconstruction directly in Python.

4 · summary construction

Builds the metrics that humans actually care about.

latency_summary_rows collect_hardware_rows make_summary_from_rows print_report

These functions compute operation deltas, op latency in milliseconds, memory and cache statistics, normalized CPU, context switching, and the final Markdown tables.

5 · input orchestration

Selects the right path for the input type.

iter_metric_files summarize_raw_ftdc parse_decoded_json_snapshots num_ejson dig dig_dotted summarize_decoded_json main

The CLI dispatcher decides whether the input is raw FTDC or decoded JSON, then routes everything into the same summary model.

Lineage

What evolved from the original scripts.

Interpretation guardrail
  • Latency rows are averages derived from cumulative MongoDB counters, not percentile histograms.
  • CPU appears only when the FTDC source actually contains `systemMetrics.cpu`.
  • System memory rows appear only when the raw FTDC stream exposes the corresponding `systemMetrics.memory` fields or supported aliases.
Key calculations

The script converts cumulative counters into usable rates.

These are the core formulas behind the report, expressed the same way the script reasons about the data.

Latency in milliseconds

window_avg_latency_ms = ((last_latency_us - first_latency_us) / (last_ops - first_ops)) / 1000

The reporter uses the delta of cumulative latency and op count to estimate the average latency within the captured window.

Throughput

ops_per_second = delta_ops / elapsed_seconds

Every op-latency row includes throughput so the latency number has volume context.

Normalized CPU

cpu_normalized_pct = ((delta_counter_ms / delta_wall_ms) * 100) / cores

The script hides raw aggregate CPU percentages and keeps only the normalized per-core view, which is easier to interpret across different host sizes.

Cache fill ratio

cache_fill_ratio_pct = (bytes_currently_in_cache / maximum_bytes_configured) * 100

This is one of the fastest ways to understand whether the WiredTiger cache is lightly used or approaching pressure.

Usage

Run it three ways.

The interface is deliberately small: one required input path, with an optional override for decoded JSON mode.

CLI examplespython3
python3 ftdc_metrics_report.py /path/to/metrics.2026-05-11T04-19-11Z-00000
